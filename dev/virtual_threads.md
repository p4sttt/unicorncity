# 1. Эволюция многозадачности в Java Virtual Machine

## 1.1. Ограничения классической модели потоков 1:1
Исторически многозадачность в платформе Java базировалась на модели прямого отображения ($1:1$) логических потоков виртуальной машины (`java.lang.Thread`) на физические потоки операционной системы (ядра ОС). Данный подход позволил делегировать планирование и распределение процессорного времени зрелым механизмам ядра ОС. Однако по мере роста масштабов веб-сервисов и смещения характера нагрузок в сторону преобладания операций ввода-вывода (I/O-bound) выявились фундаментальные архитектурные ограничения модели $1:1$:

* **Высокое потребление физических ресурсов.** Каждый поток операционной системы требует резервирования фиксированного объема памяти под стек вызовов (по умолчанию в 64-разрядных JVM HotSpot этот объем составляет 1 МБ). При попытке масштабирования системы до сотен тысяч или миллионов одновременных соединений потребление оперативной памяти только на поддержание стеков потоков становится критическим.
* **Высокая стоимость переключения контекста.** Переключение выполнения между потоками ОС сопряжено с переходом процессора из пространства пользователя в пространство ядра (kernel transition). Эта операция требует сохранения состояния регистров CPU, изменения таблиц страниц памяти и сброса кэша ассоциативной трансляции (TLB), что вносит существенные накладные расходы времени.

## 1.2. Ограничения пулов потоков при масштабировании I/O-bound задач
Для минимизации накладных расходов на создание потоков традиционно применяется абстракция пулов потоков (`ExecutorService`). Тем не менее использование пулов порождает ряд сопутствующих проблем:

* **Искусственное ограничение пропускной способности.** Лимитирование размера пула скрывает истинную конкуренцию на уровне ввода-вывода. Потоки пула блокируются на время ожидания ответа от внешних ресурсов (баз данных, API), простаивая вместо выполнения полезных вычислений. Это искусственно сдерживает утилизацию вычислительной мощности оборудования.
* **Риски некорректного управления состоянием.** Сохранение данных в контексте потока посредством `ThreadLocal` при многократном использовании потоков пула для разных задач ведет к утечкам памяти и возможности несанкционированного доступа к данным между несвязанными транзакциями.
* **Искажение семантики прерываний.** Механизм прерывания задач через `Thread.interrupt()` становится ненадежным в условиях повторно используемых потоков пула, поскольку прерывание может повлиять на последующую задачу, выполняющуюся на том же потоке, или быть проигнорировано библиотечным кодом.

# 2. Асинхронные альтернативы и их архитектурные компромиссы

## 2.1. Реактивное программирование и событийный ввод-вывод
Для преодоления барьера масштабируемости модели $1:1$ индустрия выработала реактивную парадигму, основанную на неблокирующем системном вводе-выводе (механизмы `epoll` / `kqueue` на уровне ядра) и библиотеках абстракции (например, `CompletableFuture`, Project Reactor, RxJava). В рамках этого подхода поток ОС возвращается в пул на время ожидания I/O-операции, а обработка завершения ввода-вывода делегируется событийному циклу (Event Loop).

Несмотря на высокую пропускную способность, реактивное программирование требует перехода от императивного стиля к декларативному, основанному на цепочках функций обратного вызова (callbacks), что влечет за собой критические эргономические и эксплуатационные недостатки:

* **Инверсия управления (Callback Hell).** Код теряет линейную структуру, усложняется ветвление бизнес-логики и обработка исключений. Ниже представлен пример асинхронного чтения и записи данных через `AsynchronousSocketChannel`, иллюстрирующий сильное зацепление и фрагментацию кода:

```Java
// Иллюстрация инверсии управления при асинхронном сетевом чтении
AsynchronousSocketChannel socketChannel = AsynchronousSocketChannel.open();
socketChannel.connect(address, null, new CompletionHandler<Void, Void>() {
    @Override
    public void completed(Void result, Void attachment) {
        ByteBuffer buffer = ByteBuffer.allocate(1024);
        socketChannel.read(buffer, null, new CompletionHandler<Integer, Void>() {
            @Override
            public void completed(Integer bytesRead, Void attachment) {
                buffer.flip();
                byte[] data = new byte[buffer.remaining()];
                buffer.get(data);
                processDataAsync(data, new Callback<Result>() {
                    @Override
                    public void onSuccess(Result res) {
                        ByteBuffer response = ByteBuffer.wrap(res.serialize());
                        socketChannel.write(response, null, new CompletionHandler<Integer, Void>() {
                            @Override
                            public void completed(Integer bytesWritten, Void attachment) {
                                try {
                                    socketChannel.close();
                                } catch (IOException e) {
                                    // Локальная обработка ошибок закрытия канала
                                }
                            }
                            @Override
                            public void failed(Throwable exc, Void attachment) {
                                // Локальная обработка ошибок записи
                            }
                        });
                    }
                    @Override
                    public void onFailure(Throwable t) {
                        // Локальная обработка ошибок бизнес-логики
                    }
                });
            }
            @Override
            public void failed(Throwable exc, Void attachment) {
                // Локальная обработка ошибок чтения
            }
        });
    }
    @Override
    public void failed(Throwable exc, Void attachment) {
        // Локальная обработка ошибок подключения
    }
});
```

* **Потеря контекста выполнения.** Поскольку шаги выполнения одной логической операции распределяются по разным физическим потокам планировщика, классические трассировки стека (stack traces) становятся неинформативными, отображая лишь внутренний стек событийного цикла. Это затрудняет отладку и логирование.
* **Деградация инструментария отладки и профилирования.** Становится невозможной классическая линейная отладка методом пошагового обхода (step-over). Инструменты профилирования (например, Java Flight Recorder) фиксируют ложные состояния простоя пулов потоков, скрывая реальные узкие места производительности.

## 2.2. Парадигма Continuation Passing Style (CPS) и компиляторные корутины
Альтернативным вектором развития асинхронности стало автоматическое преобразование кода компилятором в рамках модели Continuation Passing Style (CPS). В JVM-экосистеме наиболее ярким представителем данной парадигмы являются сопрограммы Kotlin. Разработчик пишет императивный код, помечая точки потенциальной приостановки ключевым словом `suspend`. Компилятор преобразует такие функции в конечные автоматы, проецируя локальные переменные в поля объекта состояния (`Continuation`), а само выполнение разделяет на сегменты внутри ветвлений `switch`.

Несмотря на сохранение линейной ментальной модели, подход CPS имеет существенные архитектурные издержки:

* **Эффект «окрашивания функций» (Function Coloring).** Асинхронный контекст является транзитивно инфекционным: приостанавливаемая функция может быть вызвана только из другой приостанавливаемой функции или специализированного билдера корутин. Это разделяет кодовую базу на два изолированных и несовместимых лагеря.
* **Снижение эффективности JIT-компиляции.** Разделение единого алгоритма на множество мелких объектов и ветвлений внутри конечного автомата препятствует применению базовых оптимизаций виртуальной машины HotSpot, таких как встраивание методов (inlining) и escape-анализ для исключения аллокаций.
* **Нагрузка на сборщик мусора (GC Pressure).** На каждом шаге приостановки выполнения в куче создаются вспомогательные объекты состояний, что увеличивает интенсивность выделения памяти и частоту циклов сборки мусора.

## 2.3. Сравнительный анализ: Stackless против Stackful корутин (C++ vs Project Loom)
Различия в реализации сопрограмм в современных языках программирования определяются их стековой архитектурой:

* **Stackless (бесстековые) корутины (C++20, Kotlin).** В C++20 корутины используют ключевые слова `co_await`, `co_yield` и `co_return`. Состояние корутины упаковывается во фрейм, аллоцируемый в куче, но сама корутина не имеет собственного стека вызовов. Вызов вложенных функций из корутины происходит на обычном физическом стеке потока. Это обуславливает жесткую необходимость в статической поддержке компилятором и приводит к эффекту окрашивания функций, так как приостановка выполнения возможна только непосредственно внутри тела корутины.
* **Stackful (стековые) корутины (Project Loom, Go).** Виртуальные потоки в Project Loom не зависят от компиляторного преобразования CPS и управляют полноценным логическим стеком вызовов на уровне виртуальной машины. Виртуальный поток может быть приостановлен в любой точке глубоко вложенного стека вызовов, включая вызовы стандартных Java-библиотек, написанных без учета асинхронности. Это полностью исключает проблему окрашивания функций и обеспечивает стопроцентную обратную совместимость с унаследованным синхронным кодом.

## 2.4. Кооперация виртуальных потоков и реактивных фреймворков
Виртуальные потоки не следует рассматривать как концепцию, полностью вытесняющую реактивное программирование. Как отмечается в исследованиях интеграции реактивного подхода и Project Loom (например, в докладе *«Project Loom — a friend or foe of Reactive?»*), данные технологии решают разные задачи и взаимно дополняют друг друга:

* **Project Loom** предоставляет эффективную и дешевую модель конкурентности для императивного кода, решая проблему блокировки потоков ОС при вводе-выводе.
* **Реактивные фреймворки** предлагают развитый декларативный инструментарий для композиции асинхронных потоков данных, гибкого управления противодавлением (backpressure), динамического объединения потоков событий, а также реализации паттернов отказоустойчивости (например, Circuit Breaker).

Интеграция этих подходов позволяет использовать виртуальные потоки в качестве легковесных исполнителей (schedulers) внутри реактивных конвейеров для обработки блокирующих вызовов, объединяя простоту императивного выполнения с мощностью реактивной оркестрации данных.

# 3. Внутренняя архитектура виртуальных потоков (Project Loom)

Project Loom интегрирует концепцию легковесной многозадачности непосредственно в рантайм HotSpot JVM, реализуя модель сопоставления $M:N$. В этой схеме $M$ виртуальных потоков планируются к выполнению на $N$ несущих (carrier) платформенных потоках ОС. Класс `java.lang.Thread` унифицирует обе реализации, скрывая детали переключения контекста под общим интерфейсом.

## 3.1. Делимитированные продолжения (Delimited Continuations) на уровне JVM
Фундаментальным механизмом виртуальных потоков является концепция ограниченных (делимитированных) продолжений, реализованная в HotSpot JVM через недокументированный класс `jdk.internal.vm.Continuation` и область видимости `ContinuationScope`.

В отличие от неограниченных продолжений (First-Class Continuations), захватывающих все состояние процесса, ограниченное продолжение представляет собой снимок стека вызовов конкретного изолированного вычисления с четко определенными границами. Операция приостановки `Continuation.yield(Scope)` осуществляет каскадный разбор стека строго до кадра, соответствующего указанному `ContinuationScope`.

Ниже приведен демонстрационный пример низкоуровневой работы с продолжениями, показывающий чередование выполнения между вызывающим контекстом (host) и продолжением:

```Java
import jdk.internal.vm.Continuation;
import jdk.internal.vm.ContinuationScope;

public class ContinuationSubsystemDemo {
    private static final ContinuationScope SCOPE = new ContinuationScope("LoomCoreDemo");

    public static void main(String[] args) {
        Continuation cc = new Continuation(SCOPE, () -> {
            System.out.println("[Continuation] Шаг 1");
            Continuation.yield(SCOPE);
            System.out.println("[Continuation] Шаг 2");
            Continuation.yield(SCOPE);
            System.out.println("[Continuation] Шаг 3");
        });

        System.out.println("[Host] Вызов 1");
        cc.run();
        System.out.println("[Host] Вызов 2");
        if (!cc.isDone()) {
            cc.run();
        }
        System.out.println("[Host] Вызов 3");
        if (!cc.isDone()) {
            cc.run();
        }
    }
}
```

Используя механизм продолжений, рантайм Java позволяет конструировать генераторы ленивых вычислений без привлечения примитивов операционной системы:

```Java
import jdk.internal.vm.Continuation;
import jdk.internal.vm.ContinuationScope;
import java.util.Iterator;
import java.util.NoSuchElementException;
import java.util.function.Consumer;

public class ContinuationGenerator<T> implements Iterable<T> {
    private static final ContinuationScope GEN_SCOPE = new ContinuationScope("GeneratorScope");
    private final Consumer<GeneratorContext<T>> lambdaTarget;

    public ContinuationGenerator(Consumer<GeneratorContext<T>> lambdaTarget) {
        this.lambdaTarget = lambdaTarget;
    }

    public static class GeneratorContext<T> {
        private T nextValue;
        public void emit(T value) {
            this.nextValue = value;
            Continuation.yield(GEN_SCOPE);
        }
    }

    @Override
    public Iterator<T> iterator() {
        return new Iterator<>() {
            private final GeneratorContext<T> context = new GeneratorContext<>();
            private final Continuation continuation = new Continuation(
                GEN_SCOPE,
                () -> lambdaTarget.accept(context)
            );

            {
                continuation.run();
            }

            @Override
            public boolean hasNext() {
                return !continuation.isDone();
            }

            @Override
            public T next() {
                if (continuation.isDone()) {
                    throw new NoSuchElementException();
                }
                T value = context.nextValue;
                continuation.run();
                return value;
            }
        };
    }
}
```

## 3.2. Жизненный цикл стека: процессы Freeze и Thaw
С технической точки зрения HotSpot JVM решает задачу эффективного переноса активационных фреймов виртуального потока с физического стека несущего потока ОС в управляемую кучу JVM и обратно. Этот процесс полностью скрыт от прикладного разработчика и состоит из двух фаз:

1. **Freeze (Замораживание стека).** В момент, когда виртуальный поток блокируется на I/O-операции или примитиве синхронизации, рантайм инициирует вызов `Continuation.yield()`. JVM анализирует текущие фреймы на физическом стеке от текущей точки выполнения вверх до фрейма запуска продолжения `Continuation.run()`. Эти фреймы (включая локальные переменные, ссылки на объекты кучи и указатели команд `PC`) сериализуются и копируются в специальный нативный массив в куче, представленный объектом `StackChunk`. Физический стек разматывается (кадры очищаются), и управление возвращается несущему платформенному потоку.
2. **Thaw (Восстановление стека).** Когда блокирующее событие разрешается, планировщик выбирает виртуальный поток для возобновления и запускает `Continuation.run()` на свободном несущем потоке. JVM выполняет десериализацию метаданных из `StackChunk` и восстанавливает сохраненные кадры активации на вершине физического стека текущего несущего потока. Указатель инструкций `PC` восстанавливается в позицию, следующую за вызовом `yield`, обеспечивая бесшовное продолжение выполнения.

## 3.3. Ленивое копирование и барьеры возврата (Return Barriers)
Для оптимизации производительности переключения контекста в Loom применяется стратегия **ленивого копирования стека (Lazy Copying)**. При возобновлении потока (фаза Thaw) JVM не копирует весь массив `StackChunk` обратно на физический стек платформенного потока. Вместо этого копируются только самые верхние фреймы, необходимые для немедленного продолжения вычислений.

Для загрузки остальных фреймов используется механизм **барьеров возврата (Return Barriers)**. При восстановлении части стека на границе недозагруженных фреймов устанавливается специальная ловушка (Return Barrier). В момент, когда текущий метод завершает выполнение и управление должно вернуться к вышележащему (еще не скопированному) кадру, происходит перехват управления виртуальной машиной. JVM динамически догружает необходимую порцию фреймов из кучи на физический стек. Это позволяет поддерживать константное время переключения контекста, не зависящее от общей глубины стека вызовов.

# 4. Анализ ограничений и эволюция компромиссов платформы

Внедрение виртуальных потоков в экосистему Java сопряжено с рядом компромиссов, обусловленных системной архитектурой JVM. Ниже систематизированы основные ограничения и их текущий статус в современных версиях платформы (JDK 21 - JDK 26).

## 4.1. Специфика планирования и отсутствие временного вытеснения (Preemption)
* **Внутренний планировщик:** Монтирование виртуальных потоков осуществляется на базе внутренней реализации `ForkJoinPool`, функционирующей в режиме FIFO с алгоритмом воровства задач (work-stealing). В текущих версиях JDK публичный API для подключения кастомных планировщиков отсутствует. Параметры пула настраиваются исключительно через системные свойства запуска JVM (например, `jdk.virtualThreadScheduler.parallelism`).
* **Кооперативное планирование:** Виртуальные потоки используют исключительно кооперативную многозадачность. В JVM отсутствует планирование с квантованием времени (time-slice preemption). Если виртуальный поток выполняет длительные математические или CPU-bound вычисления без явных точек прерывания (блокирующий ввод-вывод, вызовы парковки `LockSupport.park()`, `Thread.sleep()` или `Thread.yield()`), он монополизирует несущий платформенный поток ОС, что может привести к исчерпанию ресурсов планировщика и деградации производительности всей системы.

## 4.2. Проблема пиннинга (Pinning) при нативных вызовах
При выполнении виртуальным потоком нативного кода через интерфейсы JNI (Java Native Interface) или современный Project Panama (Foreign Function & Memory API) возникает эффект **прикрепления (pinning)** к несущему платформенному потоку. Поскольку виртуальная машина не контролирует стек вызовов внешнего нативного кода (C/C++), она не способна безопасно выполнить сериализацию этих фреймов в кучу. В результате виртуальный поток намертво блокирует подлежащий поток ОС на все время нативного вызова, лишая планировщик возможности переиспользовать этот поток для других задач. Для идентификации таких зон рекомендуется использовать события профилирования `jdk.VirtualThreadPinned` в Java Flight Recorder.

## 4.3. Устранение пиннинга для синхронизации мониторов (JEP 491)
В ранних версиях Project Loom (включая JDK 21) использование базовых примитивов синхронизации мониторов (блоки `synchronized` и методы с этой инструкцией) приводило к пиннингу несущего потока. Это было связано с тем, что нативная структура `ObjectMonitor` в HotSpot JVM жестко привязывалась к идентификатору системного потока ОС (`Thread ID`), и рантайм не мог безопасно выполнить `yield` во время удержания монитора.

Эта проблема была фундаментально решена в рамках **JEP 491 (Synchronize Virtual Threads without Pinning)**, интегрированного в JDK 24 и ставшего стандартным поведением в последующих версиях (JDK 25, 26):

* Нативная реализация `ObjectMonitor` была реорганизована и отвязана от жесткой ассоциации со стеком конкретного платформенного потока ОС.
* Теперь, когда виртуальный поток блокируется на закрытом мониторе внутри блока `synchronized` или вызывает `Object.wait()`, рантайм HotSpot сериализует состояние владения монитором и ассоциирует его непосредственно с объектом `Continuation` в куче.
* Операция приостановки `yield()` отрабатывает корректно: фреймы виртуального потока выгружаются в `StackChunk` в куче, а несущий поток ОС освобождается для выполнения других задач из `ForkJoinPool`. При освобождении монитора планировщик возобновляет выполнение продолжения на любом свободном carrier-потоке.

Благодаря этому отпала необходимость тотального переписывания легаси-кода для замены `synchronized` на `ReentrantLock`.

# 5. Области неопределенности и направления для дальнейшего уточнения

Для полноты архитектурной картины виртуальных потоков Java ряд областей требует дополнительного анализа и детализации в рамках развития платформы:

1. **Отсутствие публичного API планировщика.** Отсутствие возможности регистрировать кастомные планировщики ограничивает использование виртуальных потоков в сценариях, требующих сложной приоритизации задач (например, в системах реального времени или мультиарендных платформах с разделением SLA). Архитектурные подходы к решению этой проблемы в будущих релизах JDK остаются неопределенными.
2. **Механизмы автоматического вытеснения CPU-bound задач.** Текущая модель полностью возлагает ответственность за предотвращение монополизации процессора на прикладного разработчика. Требуется исследовать потенциальные пути внедрения неявного вытеснения со стороны JVM (например, интеграцию проверок прерывания в точки безопасного останова — safepoints) для защиты планировщика от CPU-интенсивных задач.
3. **Неблокирующая интеграция с нативным кодом.** Пиннинг на стыке с JNI и Project Panama остается открытой проблемой. Необходимы архитектурные изыскания по интеграции системных асинхронных API (например, `io_uring` в Linux) на уровне рантайма виртуальной машины для предотвращения блокировки несущих потоков при вызове foreign-функций.
